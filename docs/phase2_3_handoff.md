# تسليم المرحلة P2.3 — الخطط واختبار التجاوز

التاريخ: 2026-07-12

## المنفذ

- إصلاح شاشة خطط Android وإزالة الخطأ البنيوي السابق.
- إنشاء وتعديل وحذف خطط أسبوعية وشهرية في Android والويب.
- أزرار زيادة ونقصان لمقرر الحفظ والمراجعة من النموذج والبطاقة.
- تحديث مقرر الطالب الافتراضي عند تفعيل خطة نشطة أو تعديلها.
- نقل خطط الويب من الذاكرة المؤقتة إلى Supabase.
- إكمال الخطة يجعلها بانتظار اختبار تجاوز.
- منع الخطة التالية حتى ربط اختبار ناجح للطالب بعد إكمال الخطة السابقة.
- طباعة A4 وكاشير 80 مم مع جدول الأيام واستثناء العطلة الأسبوعية.
- مزامنة ثنائية الاتجاه بأولوية `updated_at` ومنع عودة الحذف بين الويب والهاتف.

## ملف SQL المطلوب

الملف الرئيسي الخاص بهذه المرحلة هو:

`website/supabase/migrations/20260712000100_p5_smart_plans.sql`

ولمن نزّل النسخة القديمة التي فشلت، يوجد ملف تمهيدي صغير يمكن تشغيله أولًا ثم إعادة تشغيل ملف الخطط القديم:

`website/supabase/migrations/20260712000050_scope_helpers_compat.sql`

هذا migration تراكمي ولا يحتوي على `DROP TABLE`. يضيف الحقول والقيود والفهرس والمشغل وسياسة RLS اللازمة. لا تشغّل `website/database_schema.sql` على قاعدة قائمة لأنه ملف تأسيس قديم يحتوي أوامر حذف.

## ترتيب التطبيق المقترح

1. خذ نسخة احتياطية من Supabase.
2. نفّذ ملف SQL المصحح الخاص بهذه المرحلة كاملًا؛ أصبح ينشئ دوال نطاق الصلاحيات إذا كانت القاعدة لا تحتويها.
3. إذا ظهر سابقًا خطأ `current_user_can_access_halaqa does not exist` فلا يلزم تراجع يدوي؛ كان الملف داخل transaction وقد تراجعت العملية تلقائيًا.
4. اختبر بحساب مدير وبحساب معلم أن كل مستخدم يرى خطط حلقته فقط.
5. أنشئ خطة، أكملها، وتأكد أن الخطة التالية محظورة.
6. سجّل اختبارًا ناجحًا بعد الإكمال، واربطه بالخطة، ثم أنشئ الخطة التالية.
7. بعد نجاح القاعدة التجريبية فقط، كرر التنفيذ على الإنتاج.

> بناءً على مخطط القاعدة المرفق، ما تزال migration سلامة تقدم الطالب `20260711000100_p0_student_progress_integrity.sql` مطلوبة لإضافة حقول المحفوظ وQR ونطاق الحلقة. طبّقها منفصلة بعد النسخة الاحتياطية وقبل اختبار المزامنة الشامل.

## استعلامات التحقق بعد التنفيذ

```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'public'
  AND table_name = 'plans'
  AND column_name IN (
    'halaqa_id', 'test_status', 'completion_exam_id',
    'completed_at', 'deleted_at', 'updated_at'
  )
ORDER BY column_name;

SELECT policyname
FROM pg_policies
WHERE schemaname = 'public' AND tablename = 'plans';

SELECT trigger_name
FROM information_schema.triggers
WHERE event_object_schema = 'public'
  AND event_object_table = 'plans';
```

## اختبار القبول على الجهاز

1. أعد فتح التطبيق بعد إنشاء الخطة وتأكد أنها لم تختفِ.
2. جرّب زيادة وإنقاص الآيات والصفحات والأسطر من الخطة.
3. عدّل الخطة واطبعها A4 ثم كاشير 80 مم.
4. أكمل الخطة وحاول إنشاء أخرى قبل الاختبار؛ يجب المنع.
5. سجّل اختبارًا بدرجة أقل من 60؛ يجب ألا يُقبل.
6. سجّل اختبارًا ناجحًا بعد الإكمال واعتمده؛ يجب فتح الخطة التالية.
7. عدّل الخطة من الويب ثم زامن Android وتأكد أن أحدث تعديل هو الظاهر.

لم يُنفذ SQL على قاعدة Supabase، ولم تُعدّل بيانات إنتاجية من بيئة التطوير.
