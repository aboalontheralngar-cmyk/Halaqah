# تسليم المرحلة P6.2 — حماية البيانات والنسخ السحابي المشفر

التاريخ: 2026-07-13

## النتيجة

أصبحت النسخة الاحتياطية ملفًا مشفرًا لا JSON مكشوفًا، ويمكن حفظها محليًا أو
رفعها إلى Supabase Storage ثم استعادتها بعد التحقق من عبارة الحماية وسلامة
المحتوى. كما أضيف سجل تدقيق محلي وسحابي وصفحات خصوصية واحتفاظ في Android
والويب.

## عطل الاستعادة المصحح

كان التطبيق يكتب `version: 2.1` لكنه يقبل `2.0` فقط عند الاستيراد. أصبحت
الاستعادة تقبل `2.0` و`2.1` و`3.0` والنسخ القديمة، وتكتب النسخ الجديدة بصيغة
`3.0` المشفرة فقط.

## نموذج الحماية

- AES-256-GCM للتشفير والتحقق من العبث.
- PBKDF2-HMAC-SHA256 بعدد 600,000 دورة لاشتقاق المفتاح.
- ملح عشوائي 16 بايت وnonce جديد لكل ملف.
- SHA-256 داخل المحتوى كتحقق إضافي بعد فك التشفير.
- عبارة الحماية في Android KeyStore عبر `flutter_secure_storage`، ولا تدخل
  SQLite أو ملف النسخة أو سجل التدقيق.
- حد أقصى 100MB وفحص كامل قبل بدء معاملة الاستعادة.
- الحد الأدنى Android 6.0 (API 23)، مع تعطيل Android Auto Backup العام.

## النسخ السحابي

- المزود المختار: Supabase Storage الموجود أصلًا في المشروع، بدل إضافة OAuth
  ومزود جديد قبل الإطلاق.
- bucket خاص باسم `halaqah-backups`، والملف مشفر قبل مغادرة الجهاز.
- المسار يبدأ بمعرف المستخدم ثم المركز، وسياسات Storage تمنع حسابًا من رؤية
  مجلد حساب آخر.
- احتفاظ 10 أو 30 أو 60 نسخة، وحذف الأقدم بعد نجاح الرفع فقط.
- فشل الرفع لا يحذف النسخة المحلية ولا يرجع عملية النسخ.

## سجل التدقيق والخصوصية

- SQLite v14: جدول `audit_events` ومحفزات على الطلاب والحضور والتسميع والنقاط
  والإجازات والخطط والاختبارات والعائلات وغيرها.
- Supabase: جدول غير قابل للتعديل للمستخدم العادي، RPC بهوية `auth.uid()`،
  ومحفزات لا تحفظ إلا نوع العملية والمعرف وأسماء الحقول المعدلة.
- Android: شاشة سجل تدقيق، إعداد مدة الاحتفاظ، وسياسة خصوصية.
- Web: `/audit-log` و`/privacy` وروابط من الإعدادات والقائمة الجانبية.

## SQL المطلوب

نفّذ محتوى الملف، لا اسمه:

`website/supabase/migrations/20260713000300_p6_data_privacy_cloud_backup.sql`

الملف مكتفٍ ذاتيًا ويعيد إنشاء
`current_user_can_access_halaqa(uuid, uuid)`، ولا يشير إلى
`progress.halaqa_id`، ولا يحتوي `DROP TABLE` أو `TRUNCATE`.

## اختبار القبول قبل الإنتاج

1. أخذ نسخة من قاعدة Supabase الحالية.
2. تطبيق migration على مشروع مرحلي.
3. إعداد عبارة حماية في Android وإنشاء نسخة محلية وسحابية.
4. التأكد أن الملف لا يحتوي اسم طالب عند فتحه كنص.
5. تنزيله على جهاز ثانٍ وتجربة عبارة خاطئة ثم الصحيحة.
6. مقارنة عدد الطلاب وآخر حفظ ومراجعة وخطة قبل الاستعادة وبعدها.
7. تسجيل دخول معلم حلقة أخرى والتأكد أنه لا يرى أحداث أو بيانات الحلقة.
8. عدم تفعيل النسخ السحابي في الإنتاج حتى تنجح الخطوات السابقة.
